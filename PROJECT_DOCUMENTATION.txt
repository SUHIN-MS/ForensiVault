ForensiVault Project Documentation

0. Technical Overview

ForensiVault is built on a layered tech stack that starts with familiar web technologies and extends into a complex forensic analysis engine. Developers and newcomers should understand the components in order of increasing complexity:

- **Core Technologies**: Node.js (≥16) with Express, MongoDB, Python 3.10+ with Flask, React/Vue frontend, JWT for auth, AES-256-CBC for encryption, multer for uploads, and pytsk3 for disk parsing.
- **Directory Layout**: the repository separates concerns – `server.js` for backend logic, `models/` for Mongoose schemas, `services/` and `utils/` for helpers, `python-services/` for forensic modules, and `forensivault-ui/` for UI code.
- **Python Forensics Service**: handles low‑level disk image parsing, file extraction, content searching, and carving. Modules include `disk_parser.py`, `file_extractor.py`, `content_analyzer.py`, `file_carver.py`, and orchestration in `app.py` with Flask and threading.
- **Node-to-Python Bridge**: the `pythonBridge` utility makes HTTP calls to the Python service (e.g. `parseDiskImage`, `extractFile`) from Node routes, enabling transparent cross-language integration.
- **Security & Tamper Detection**: JWT-based auth, role middleware, AES encryption for file storage, SHA-256 hashing, immutable audit logs using a hash chain, and content comparison routines for detecting tampering (text diffs, image metadata, binary checksums).
- **Complex Features**: background carving threads, hash chain verification jobs, extensible signature lists, OCR toggling, and a full REST API consumed by the frontend.
- **Extensibility Tips**: add new filesystems or file types, enable OCR with Tesseract, modify frontend independently of backend, or swap MongoDB if needed.

This roadmap lets beginners start with the basics (install prerequisites, run servers) and gradually explore the more complex forensic analysis and security mechanisms built into the system.

1. Project Overview

_For someone just walking into the room:_ ForensiVault is like a secure digital evidence locker. Imagine you find a hard drive at a crime scene. Instead of opening it and risking changes, you make a copy (a disk image), then you can look inside that copy without ever touching the original. ForensiVault helps you:

- **Upload and store** that disk copy safely.
- **Look at the files** inside as if they were on a normal computer, even if they're deleted.
- **Pull files out** and keep a fingerprint (hash) of every file so you know later if anyone changed it.
- **Check again later** to see if the file has been tampered with; if it has, the system raises an alarm.
- **Keep track of everything** that happens – every login, upload, change – in a ledger that can't be altered.
- **Talk to all parts** via simple web requests (API endpoints) so other apps or a webpage can use it.
- **Ask a helper program** written in Python to do the heavy lifting of reading raw disk data.

At its heart, ForensiVault is made for forensic investigators who need to prove that digital evidence stayed the same from the moment it was collected until it's used in court.

For newbies, think of it as Dropbox + detective tools + a tamper alarm, all wrapped in code.

For techies, it's a Node/Express backend, Python/Flask parsing service, MongoDB storage, and a React/Vue frontend.

**Glossary & Terminology**

- **Evidence:** Any digital file/image collected during an investigation.
- **Disk image:** A raw sector-by-sector copy of a storage device. Treated as read-only input to parsers and carvers.
- **Tampering:** Any modification of evidence after its initial hash is recorded. The system detects changes by re-hashing extracted content and comparing metadata.
- **Hash chain:** Cryptographic chain used in logs where each entry includes the previous entry's hash, preventing undetected edits.
- **Parser:** Component (usually Python) that interprets filesystem structures inside a disk image.
- **Carver:** Tool that scans raw data for known file signatures to recover deleted or unreferenced files.
- **JWT:** JSON Web Token used for stateless authentication.
- **Multer:** Node middleware for handling file uploads.

These terms appear repeatedly throughout the documentation, so understanding them will make the rest of the guide easier to follow.

2. Architecture

The system is composed of:

- Node.js server (Express) performing authentication, authorization, tamper detection logic, and API endpoints.
- Python Flask service handling disk image parsing, file extraction, searching, statistics, and carving using pytsk3 and other forensic libraries.
- MongoDB database storing users, cases, evidences, logs, and disk image metadata.
- Frontend application consuming the REST API to provide a user interface.

Components communicate via HTTP; the Node server uses a custom pythonBridge module to call the Python service.

4. Setup & Execution

Prerequisites:
- Node.js ≥16, Python 3.10+, MongoDB
- OS packages required for pytsk3 (The Sleuth Kit) such as `libtsk-dev` on Linux. On Windows, use precompiled wheels or a Docker container.

**Step-by-Step Install**

1. **Clone repository** and enter it:
    ```bash
    git clone https://github.com/yourorg/forensivault.git
    cd forensivault
    ```
2. **Configure `.env`** – copy example and update values:
    ```bash
    cp .env.example .env
    # edit .env:
    # MONGODB_URI=mongodb://localhost:27017/forensivault
    # JWT_SECRET=someLongRandomString
    # AES_KEY=<32-byte hex> AES_IV=<16-byte hex>
    # CLIENT_URL=http://localhost:5173
    # PYTHON_SERVICE_URL=http://127.0.0.1:5001
    ```
3. **Install Node dependencies**:
    ```bash
    npm install            # installs express, mongoose, multer, etc.
    ```
    *Expected output:* `added 259 packages`… no errors.
4. **Start Node server (dev mode)**:
    ```bash
    npm run start          # runs node server.js
    ```
    *You should see* `✅ MongoDB Connected` and `Server listening on port 3000`.

5. **Setup Python service**:
    ```bash
    cd python-services
    python -m venv venv
    .\venv\Scripts\activate      # or source venv/bin/activate
    pip install -r requirements.txt
    python app.py
    ```
    *Output:* `* Running on http://127.0.0.1:5001/ (Press CTRL+C to quit)`
6. **Verify connectivity**:
    ```bash
    curl http://127.0.0.1:5001/health
    # {
    #   "status": "running",
    #   "service": "ForensiVault Python Services",
    #   "version": "1.0.0",
    #   "timestamp": "2026-03-01T12:00:00",
    #   "capabilities": {"disk_parsing": true, "file_extraction": true, "content_search": true, "file_carving": true, "ocr": false}
    # }
    ```

7. **Start frontend (optional)**
    ```bash
    cd ../forensivault-ui
    npm install
    npm run dev            # serves at localhost:5173
    ```

With all three servers running, the application is ready.

6. Security & Tamper Detection

**Authentication & Authorization**
- JWT tokens signed with `JWT_SECRET`; expiry `24h`. `authMiddleware` checks header or `?token` query.
- `roleMiddleware(...roles)` restricts operations; admin-only routes manage users/cases.

**Encryption & Hashing**
- File uploads stored temporarily then encrypted with AES‑256‑CBC. Keys: `AES_KEY` (32 bytes hex) and `AES_IV` (16 bytes hex) from .env.
- `encryptFile` / `decryptFile` stream data to avoid loading entire files.
- `hashFile(filePath)` reads in chunks, updates a SHA‑256 hasher, returns hex digest.

**Immutable Audit Logs**
- `createLog(action, userId, details)` saves a `Log` document with fields:
  ```js
  { index, action, userId, details, payload, previousHash, entryHash, timestamp }
  ```
  `payload` is JSON stringified details+timestamp. `entryHash = SHA256(prevHash + payload)`.  
  The chain starts with `prevHash = "GENESIS"`.
- Queries to logs can verify integrity by re‑computing hashes.

**Tampering Detection Flow**
1. During upload or extraction, server calls `extractContent(filePath,mimeType)` which returns `contentType`, `contentHash` and metadata.  
2. Evidence record saved with `originalContent = result` and `hash = contentHash`.
3. Later, call `analyzeFile` or periodic job → `extractContent` on current file.  
4. `compareContent(orig,current)`:
   - If text: line-by-line diff with added/removed/modified counts.
   - If image: compare width/height/format/size.
   - Else: compare checksums and size.
   Severity classification (`low/medium/high/critical`) based on percentage changes.
5. On change, server updates `evidence.tamperingHistory` array and logs `TAMPERING_DETECTED` with details.
6. Evidence access may be blocked and case notified.

**Hash Chain Verification**
- A nightly job can iterate `Log.find().sort({index:1})` and recompute `entryHash` vs stored, alerting if mismatch (indicating log tampering).


7. Content Extraction & Comparison

`services/contentExtractor.js` implements:

- extractContent(filePath, mimeType): detect type (text/image/binary) and return metadata/hashes.
- compareContent(originalData, currentData): route to compareTextContent, compareImageMetadata or compareBinaryContent; calculate diffs, severity.

9. Python Service Internals

The Python service exposes a rich Flask API for low‑level disk operations.

**config.py**
- Defines directories: `BASE_DIR`, `EXTRACTED_DIR`, `CARVED_DIR`, `TEMP_DIR` etc.
- Sets limits: `MAX_DISK_IMAGE_SIZE = 1 * 1024**4` (1 TB).
- Allowed extensions for disk images and searchable files.
- `ensure_directories()` creates required folders.

**disk_parser.py (DiskImageParser)**
- Wraps `pytsk3.Img_Info` and `pytsk3.FS_Info`.
- Auto-detects partition offsets by trying common boundaries (2048, 63, etc.).
- Methods:
    - `get_disk_info()` returns size, filesystem type, sector size, partition table details.
    - `list_files(directory, recursive)` returns list of `{'path','name','inode','size','allocated','meta'}`.
    - `get_file_tree(max_depth)` builds nested dict structure with lazy evaluation.
    - `get_file_info(file_path)` returns metadata with timestamps, hashes if computed.
    - `get_statistics()` computes counts per category, top-large files, extension distribution.
    - `close()` closes the underlying `Img_Info` to release file descriptors.

Example class skeleton:
```python
class DiskImageParser:
    def __init__(self, image_path):
        self.img = pytsk3.Img_Info(image_path)
        self.fs = pytsk3.FS_Info(self.img)
        self.open_files = {}

    def get_disk_info(self):
        return {
            'size': self.img.get_size(),
            'block_size': self.fs.info.block_size,
            'fs_type': self.fs.info.ftype,
        }

    def list_files(self, directory='/', recursive=False):
        results = []
        directory_obj = self.fs.open_dir(path=directory)
        for entry in directory_obj:
            if entry.info.name.name in [b'.', b'..']:
                continue
            fname = entry.info.name.name.decode('utf-8', errors='ignore')
            path = os.path.join(directory, fname)
            results.append({
                'path': path,
                'size': entry.info.meta.size,
                'allocated': bool(entry.info.meta.flags & pytsk3.TSK_FS_META_FLAG_ALLOC),
            })
            if recursive and entry.info.meta.type == pytsk3.TSK_FS_META_TYPE_DIR:
                results.extend(self.list_files(path, recursive=True))
        return results
```

This snippet illustrates how parser wraps pytsk3 and exposes simple Python methods used by Flask endpoints.

**file_extractor.py (FileExtractor)**
- Accepts a `DiskImageParser` instance.
- `get_preview(file_path, max_size)` reads first `max_size` bytes, returns base64 or text snippet.
- `extract_file(file_path, output_dir)` streams chunks (1 MB) to disk, computing hashes simultaneously.
- `compute_hash(file_path, algorithms)` returns dict of requested hashes.
- Handles errors gracefully by returning `{success:False,error:...}`.

**content_analyzer.py (ContentAnalyzer)**
- Indexes file metadata from parser to allow fast searches.
- `search(...)` filters by filename/path/extension/type/size/date.
- `search_content(query, extensions, case_sensitive, max_results)` opens each file (binary or text), optionally runs OCR if `Config.OCR_ENABLED`, and returns matches with context.

**file_carver.py (FileCarver)**
- Reads raw image sectors, looks for signatures defined in `SIGNATURES` dictionary.
- `carve(output_dir, file_types=None, progress_cb=None)` scans sequentially, writes carved files with incremental names, invokes callback with percentage and status.
- `mark_deleted_files(results, parser)` cross-references carved files against parser listing to flag which were deleted/unallocated.
- Supports background threading (used by `/carve/start` in `app.py`).

**forensic_analyzer.py**
- Higher-level analysis combining extraction, hashing, timeline generation, and correlation with external data.
- Used by advanced endpoints or external scripts.

**app.py**
- Sets up Flask with CORS.
- Maintains `active_parsers` and `carving_jobs` dictionaries in memory. Both keyed by image path and cleaned on `/disk/close` or job completion.
- Defines helper functions `error_response()` and `success_response()` for consistent JSON format.
- Each endpoint wraps logic in `try`/`except` and prints traceback on error.
- Endpoints already enumerated under section 10; they simply call appropriate module methods and return results.
- `/carve/start` spawns a daemon thread to avoid blocking the Flask request.

This service runs independently; Node.js interacts with it via the bridge.


10. REST API Endpoints

The Node.js server exposes the following primary endpoints. Every route returns JSON with a `success` boolean and either `data` or `error`.

**Authentication**
- `POST /api/auth/signup`  
  **Body:** `{ username, password, role?, email?, fullName? }`  
  **Returns:** newly created user (id, username, role).
- `POST /api/auth/login`  
  **Body:** `{ username, password }`  
  **Returns:** JWT token, user info, message.
- `GET /api/auth/me`  
  **Headers:** `Authorization: Bearer <token>`  
  **Returns:** current user record (password omitted).
- `POST /api/auth/logout`  
  **Headers:** auth token  
  **Action:** writes a LOG entry and invalidates nothing (stateless).

**User Management** (admin only via `roleMiddleware('admin')`)
- `GET /api/users`  
  **Query params:** `status`, `role`, `search`, `page`, `limit`  
  **Response:** `{ users: [...], pagination: { total, page, limit, pages } }`.
- `GET /api/users/:id`  
  **Returns:** user by id or 404.
- `POST /api/users`  
  **Body:** `{ username, password, role?, email?, fullName?, department?, phone? }`  
  **Action:** creates a new user and logs `USER_CREATED`.
- `PUT /api/users/:id`  
  **Body:** updatable fields (`role,email,fullName,status,department,phone`)  
  **Action:** updates user and logs `USER_UPDATED`.
- `DELETE /api/users/:id`  
  **Action:** deletes user, forbidden if self; logs `USER_DELETED`.

**Disk & File Operations**
- `POST /api/disk/upload`  
  **Uses multer** to accept `multipart/form-data` with field `diskImage`. Stored in `disk-images/` with optional encryption. Creates a DiskImage document and returns its id.
- `POST /api/disk/parse`  
  **Body:** `{ imagePath }`  
  **Body validation:** ensures file exists, extension allowed, size <1TB. Calls Python bridge `parseDiskImage`. Response example:
  ```json
  {"success":true,"data":{"imagePath":"...","diskInfo":{"fileSystem":"NTFS","size":...}},"message":"Disk image parsed successfully"}
  ```
- `POST /api/disk/files`  
  **Body:** `{ imagePath, directory?, recursive? }` lists files.
- `POST /api/disk/tree`  
  **Body:** `{ imagePath, maxDepth? }` returns nested tree structure.
- `POST /api/disk/file/info`  
  **Body:** `{ imagePath, filePath }` returns metadata (size, timestamps, allocated).
- `POST /api/disk/file/preview`  
  **Body:** `{ imagePath, filePath, maxSize? }` returns base64 preview or plaintext snippet.
- `POST /api/disk/file/extract`  
  **Body:** `{ imagePath, filePath, outputDir? }` writes file to local `extracted/` and returns path+hash.
- `POST /api/disk/file/hash`  
  **Body:** `{ imagePath, filePath, algorithms? }` computes SHA256/MD5/SHA1 hashes.
- `POST /api/disk/search`  
  **Body:** many filters (`query`,`searchIn`, `fileTypes`,`extensions`,`minSize`,`maxSize`,`dateFrom`,`dateTo`) returns matching file list.
- `POST /api/disk/search/content`  
  **Body:** `{ imagePath, query, extensions?, caseSensitive?, maxResults? }` searches text within files; uses `ContentAnalyzer`.
- `POST /api/disk/stats`  
  **Body:** `{ imagePath }` returns statistics (counts per file type, totals, size distribution).
- `POST /api/disk/close`  
  **Body:** `{ imagePath }` instructs Python service to drop parser instance and free memory.

**File Carving**
- `GET /carve/types`  
  No auth required. Returns a JSON list of supported file signatures (e.g. JPEG, PNG, PDF etc.).
- `POST /carve/start`  
  **Body:** `{ imagePath, outputDir?, fileTypes? }` starts background carving thread. Response: `{ status:"scanning" }`. Carving job stored in memory `carving_jobs`.
- `POST /carve/status`  
  **Body:** `{ imagePath }` returns job progress object (`status`, `progress`, `filesFound`, etc.).
- `POST /carve/results`  
  **Body:** `{ imagePath }` returns completed results array.

**Tamper Analysis**
- `POST /analyze/file`  
  **Body:** `{ filePath }` reads an existing evidence file, runs `extractContent` + `compareContent` vs. stored baseline. Returns detailed diff with severity.
- `POST /analyze/batch`  
  **Body:** `{ filePaths: [...] }` processes multiple files.

**Cases & Evidence**
- `POST /api/cases` / `PUT /api/cases/:id` / `GET /api/cases` etc.  
  Case routes create a case number (`FV-YYYY-XXXX` via `generateCaseNumber()`), attach evidences, and handle case-level authorization.
- Evidence routes (`/api/evidence`) handle file uploads, hash computation, tampering metadata (`tamperingHistory` array), and status flags (`verified`, `tampered`).

**Logging**
- `GET /api/logs`  
  Query params: `action`, `userId`, `from`, `to`, `page`, `limit`.  
  Returns paginated log entries with `payload`, `entryHash`, `previousHash`.

All non‑public routes are protected by `authMiddleware`; many require `roleMiddleware` for fine‑grained permission.

13. Typical Workflow Example

1. Investigator logs in, obtains JWT.
2. Upload disk image → encrypted storage.
3. Parse image via Python service.
4. Browse file tree, preview/extract files.
5. Extracted files become evidence records; baseline hashes saved.
6. Periodically run tamper checks; modifications produce logs, alerts.
7. Run carving to recover deleted files and include them as evidence.

1. **Tamper detection helper (server.js)**
```js
async function analyzeEvidence(evidenceId) {
  const evidence = await Evidence.findById(evidenceId);
  if (!evidence) throw new Error('Not found');

  const currentPath = path.join(__dirname, evidence.storedPath);
  const currentData = extractContent(currentPath, evidence.mimeType);
  const result = compareContent(evidence.originalContent, currentData);

  evidence.tamperingHistory.push({
    detectedAt: new Date(),
    result,
  });
  if (result.severity !== 'low') {
    evidence.status = 'tampered';
  }
  await evidence.save();
  await createLog('TAMPERING_CHECK', req.user.id, {
    evidenceId,
    severity: result.severity,
  });
  return result;
}
```

3. **Python carving job thread (python-services/app.py)**

@app.route('/carve/start', methods=['POST'])
def carve_start():
    image_path = request.json['imagePath']

18. Full File-by-File System Walkthrough (Large to Small)

This section is the end-to-end “single-source-of-truth” for every major file and component in this repository.

18.1 Top-Level Architecture (the large system)
- The system has three primary layers:
  1. **Web/API layer**: Node/Express app in `server.js` handles authentication, authorization, evidence management, tamper detection, report generation, and database interaction.
  2. **Forensic processing layer**: Python Flask service in `python-services/app.py` with disk parser, extraction, search, carving, and file analysis modules.
  3. **Data layer**: MongoDB with Mongoose models (`models/Evidence.js`, `models/Log.js`, plus `User` and `Case` models referenced by server code).
- A frontend (`forensivault-ui/`) consumes the API and provides investigator UI.
- All components communicate via REST and JSON; Node and Python communicate via the `utils/pythonBridge.js` wrapper.

18.2 Core server (server.js) - complete behavior
- Imports: Express, Mongoose, Multer, fs/path/crypto, JWT, bcryptjs, cors, PDFKit, dotenv.
- Connects to MongoDB via `process.env.MONGODB_URI`.
- Uses in-memory encrypted keys from env: `JWT_SECRET`, `AES_KEY` (32 bytes), `AES_IV` (16 bytes) with runtime fallback if absent.
- Creates runtime folders automatically: `uploads`, `encrypted`, `disk-images`, `extracted`, `temp`, `reports`.
- Middleware:
  - `authMiddleware` reads `Authorization: Bearer <token>` or `?token` and verifies JWT.
  - `roleMiddleware(...roles)` enforces role-based access.
- Helpers:
  - `encryptFile`, `decryptFile` (AES-256-CBC stream encryption).
  - `hashFile` (SHA-256 stream hashing).
  - `createLog` (immutable log chain with `previousHash` + `entryHash`).
  - `generateCaseNumber`, `formatFileSize`.
- Authentication endpoints: signup/login/me/logout with encoded JWTs, password hashing via bcrypt.
- User management endpoints with role restrictions and logging.
- Disk endpoints for upload, parse, file listing, file info, file preview, extraction, hashing, searching, stats, close.
- Tamper detection endpoints and logic:
  - Uses `extractContent` and `compareContent` from `services/contentExtractor.js`.
  - Tamper checks update `Evidence.tamperingHistory` and set `status='tampered'` when needed.
- Report endpoints (PDF generation): chain of custody, case report, tampering report.
- File upload pipeline: multer stores on disk; extracted evidence may be encrypted and saved with metadata.
- Error handling and production app listener at `PORT || 3000`.

18.3 Models
- `models/Evidence.js`: `filename`, `hash`, `uploader`, `timestamp`, with default timestamp. (Note: simplified in code; app assumes richer fields for tamper history etc.)
- `models/Log.js`: `action`, `evidenceId`, `timestamp`.

18.4 Forensic content matching module (`services/contentExtractor.js`)
- Extractors:
  - `extractTextFile`: lines, words, character count, content hash.
  - `extractImageMetadata`: image width/height/format via signature bytes and CRC.
  - `extractBinaryFile`: checksums and size.
  - `extractContent(filePath, mimeType)`: chooses extraction strategy by file extension or MIME.
- Comparison functions:
  - `compareTextContent(original,current)`: line diff, counts added/removed/modified, total percent, severity.
  - `compareImageMetadata(original,current)`: compare resolution, format, size and compute severity.
  - `compareBinaryContent(original,current)`: checksum + size differences.
  - `compareContent(original,current)`: route by content type.

18.5 Node <-> Python wrapper (`utils/pythonBridge.js`)
- Uses built-in Node `http` module (no fetch dependency).
- `pythonRequest(endpoint, method, body)`: JSON request with 30s timeout and helpful error messages.
- Exposes all forensic endpoints: parseDiskImage, listFiles, getFileTree, getFileInfo, getFilePreview, extractFile, computeFileHash, searchFiles, searchContent, getDiskStats, closeDiskImage, startCarving, getCarvingStatus, getCarvingResults, getSupportedCarvingTypes, analyzeFile, analyzeBatch.
- `checkPythonService()` hits `/health` and returns status.

18.6 Python service config (`python-services/config.py`)
- Directory constants: disk-images/extracted/temp/carved.
- Limits: 1TB disk image, 100MB extraction, 10KB preview.
- Allowed disk image extensions: `.img`, `.raw`, `.dd`.
- Searchable file extensions and OCR toggling.
- `ensure_directories()` creates runtime directories.

18.7 Disk parser (`python-services/disk_parser.py`)
- Uses `pytsk3` when available; else fallback parser.
- Opens disk image with `pytsk3.Img_Info` and `FS_Info`, tries offsets.
- Methods:
  - `get_disk_info()`: size, fs type, block counts.
  - `list_files(directory, recursive)`: walk directories with safe decoding.
  - `get_file_info(file_path)`: metadata and timestamps.
  - `read_file(file_path, max_size)`: random read in chunks.
  - `get_file_tree(max_depth)`: nested tree with sorted children.
  - `get_statistics()`: totals, by extension/category, largest/recent files.
- Fallback parser returns limited metadata when pytsk3 is missing.

18.8 File extraction (`python-services/file_extractor.py`)
- Extracts from disk image and writes to local output.
- Enforces maximum extraction size and directory-only checks.
- Computes hashes: md5/sha1/sha256.
- Preview generation: text decode, image base64, binary hex dump.
- MIME detection with `magic` optional fallback signatures.

18.9 Content search (`python-services/content_analyzer.py`)
- Builds in-memory index (list_files recursive).
- `search(...)`: filters by filename/path/extension/category/size/date.
- `search_content(...)`: reads file content and searches query text; supports text, PDF (PyPDF2 optional), DOCX (python-docx optional).
- Returns context snippets and match counts.

18.10 File carving (`python-services/file_carver.py`)
- Signature definitions for JPEG/PNG/GIF/BMP/PDF/ZIP/RAR/MP3/AVI/TIFF.
- `FileCarver.carve(...)`: two-phase scanning and extraction; chunked read 4MB.
- Determine bounds via footer or size fields; validate each type.
- Saves carved files with metadata and hashes.
- `mark_deleted_files(...)`: cross-reference carved hashes against active filesystem files to determine deleted evidence.
- `supported_types()` returns carving capabilities.

18.11 Forensic analyzer (`python-services/forensic_analyzer.py`)
- `ForensicAnalyzer.analyze(file_path)` returns file analysis report.
- Detects type by magic bytes and text detection.
- Type-specific checks for JPEG/PNG/PDF/GIF/BMP/TEXT; plus generic entropy and hidden strings.
- Calculates overall severity and tampering likelihood.
- Used by `/analyze/file` and `/analyze/batch`.

18.12 Python API endpoints (`python-services/app.py`) - full list
